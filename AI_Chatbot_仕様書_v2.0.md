## 1. システム概要

### 1.1 プロダクトビジョン

工務店・リフォーム店のホームページに埋め込む、AIチャットボットです。事前のDB構築・インデックス管理を行わず、リクエスト時にテナントのHPをWeb Searchで参照して回答を生成するシンプルな構成とします。

#### チャットボットの位置づけ

チャットボットは既存の電話ボタン・LINEボタンと並ぶ集客導線（CTA）のひとつです。チャットボット内でLINEや電話へ誘導するボタンは生成しません。

| 導線種別 | 役割 | 本システムの対象 |
|---|---|---|
| 電話ボタン | 即時の問い合わせ・商談 | 対象外（既存UI） |
| LINEボタン | 継続的なコミュニケーション | 対象外（既存UI） |
| AIチャットボット | 24時間の疑問解消・ヒアリング支援 | ★ 本システム |

### 1.2 v1系からの主な変更点

| 項目 | v1.3（旧） | v2.0（新） |
|---|---|---|
| 知識獲得 | 事前クロール + Qdrant（HNSW + BM25） | Web Search（テナントHP限定）|
| DB | PostgreSQL + Qdrant + Redis | なし |
| Re-ranker | CrossEncoder（BGE-reranker-v2） | なし |
| オブザーバビリティ | Langfuse（セルフホスト） | なし（初期スコープ外） |
| RAG評価 | RAGAS | なし（初期スコープ外） |
| インフラ | Docker Compose（7コンテナ） | 最小構成（API + フロントのみ） |
| LangGraph | ReActループ（フル自律型） | 検討中（後述） |

> **設計方針**　初期バージョンはWeb Search + LLMの2コンポーネントで完結させます。スケールアップが必要になった段階でDB・インデックス構築を検討します。

## 2. システム構成

### 2.1 全体アーキテクチャ

```
[ユーザー]
    ↓ 質問入力
[埋め込みウィジェット]（HTMLに1行追加）
    ↓ メッセージ送信（HTTPS）
[APIサーバ]
    ↓ テナントHP限定でWeb Search実行
[Web Search]（テナントのHPドメインに絞った検索）
    ↓ 検索結果をコンテキストとして付与
[LLM]（Claude / OpenAI / Grok：未定）
    ↓ テキスト回答生成（SSEストリーミング）
[埋め込みウィジェット]
    ↓
[ユーザーに回答表示]
```

### 2.2 技術スタック

| レイヤー | 技術選定 | 備考 |
|---|---|---|
| フロントエンド | TypeScript + Vanilla JS | 軽量ウィジェット ≤50KB（gzip） |
| バックエンドAPI | Python 3.12 + FastAPI | SSEストリーミング対応 |
| LLM | Claude / OpenAI / Grok（未定） | Web Search統合が強いモデルを選定 |
| Web Search | 未定（後述） | テナントHPドメインに限定 |
| エージェント制御 | 検討中（後述） | LangGraph or シンプルなプロンプト制御 |
| インフラ | Docker Compose | APIサーバ + Nginxの最小構成 |

## 3. LLM選定

### 3.1 候補比較

Web Searchとの統合品質を最優先に選定します。

| 項目 | Claude（Anthropic） | GPT-4o（OpenAI） | Grok（xAI） |
|---|---|---|---|
| Web Search統合 | Tool Use経由（Anthropic提供） | Responses API内蔵 | Deep Search内蔵 |
| 検索ドメイン制限 | 可能（allowed_domains指定） | 制限方法は要調査 | 制限方法は要調査 |
| 日本語品質 | 高 | 高 | 中〜高 |
| API安定性 | 高 | 高 | 中（比較的新しい） |
| コスト | 中 | 中 | 低〜中 |
| ストリーミング対応 | あり（SSE） | あり（SSE） | あり |

> **未決**　LLMの最終選定は未定です。検索ドメイン制限の実装可否を各APIで検証した上で決定します。検索範囲をテナントHPのみに絞れることが最低条件です。

### 3.2 選定基準

1. テナントHPのドメインに検索を限定できるか
2. 日本語での回答品質
3. SSEストリーミング対応
4. APIコストとレートリミット

## 4. Web Search設計

### 4.1 実装方針（未定）

Web Searchの実装方針は現在検討中です。LLM選定と連動して決まります。

| 方針 | 概要 | メリット | デメリット |
|---|---|---|---|
| A. LLM内蔵のSearch機能 | OpenAIのResponses API・GrokのDeep Search等、LLMにビルトインされた検索を使う | 実装がシンプル・LLMと検索が密結合 | ドメイン制限の柔軟性が低い可能性 |
| B. Anthropic Web Search Tool | Claude Tool Use経由でAnthropicが提供するWeb Searchを使う | ドメイン制限（allowed_domains）が明確に指定可能 | Claude採用が前提 |
| C. Tavily等サードパーティ検索API | LLMとは独立した検索APIを呼び出してコンテキストを組み立てる | LLM非依存・ドメイン制限が確実 | 実装工数が増える・追加コスト |

> **方針B（Anthropic Web Search Tool）を暫定推奨**とします。ドメイン制限が`allowed_domains`パラメータで明確に指定できるため、テナントHPのみに検索を絞るという要件に最も合致します。ただしLLM選定と合わせて最終決定します。

### 4.2 検索範囲の制御

- 検索対象はテナントが登録したHPドメインのみに制限
- テナントごとに`allowed_domains`リストを設定ファイルまたは環境変数で管理
- ドメイン外の情報は取得しない（一般的なWeb検索は行わない）

### 4.3 検索フロー

```
1. ユーザーの質問を受信
2. テナントのallowed_domainsをシステムプロンプトまたはツールパラメータに注入
3. Web Searchを実行（テナントHPのみを対象）
4. 検索結果をコンテキストとしてLLMに渡す
5. LLMがテキスト回答を生成・ストリーミング返却
```

## 5. エージェント制御

### 5.1 設計方針（検討中）

DB・RAGパイプラインを廃止したことで、エージェントの複雑性は大幅に下がります。以下2案を検討します。

| 案 | 概要 | メリット | デメリット |
|---|---|---|---|
| A. シンプルなプロンプト制御 | システムプロンプト + Web Searchの1ショット構成。LangGraph不採用 | 実装が最速・デバッグしやすい | 複雑な分岐・状態管理が難しい |
| B. LangGraph（軽量版） | Web Searchノードのみを持つ最小限のグラフ構成 | 将来の拡張がしやすい・ツール管理が明確 | A案より実装工数が増える |

> **推奨**　初期バージョンは**案A（シンプルなプロンプト制御）**で実装します。Web Search 1ツールのみの構成であればLangGraphの恩恵は限定的です。拡張が必要になった段階でLangGraphへ移行します。

### 5.2 システムプロンプト設計

| プロンプト区分 | 管理方法 | 主な内容 |
|---|---|---|
| グローバル基盤 | コードに定数として定義 | 安全性・応答フォーマット・テキストのみ回答する指示・ハルシネーション防止 |
| 業種共通 | コードに定数として定義 | 住宅業界の専門用語・丁寧語設定・典型質問パターン |
| テナント固有 | 設定ファイル（YAML / 環境変数） | 会社名・対応エリア・強み・NGワード・トーン・検索対象ドメイン |

## 6. フロントエンド仕様

### 6.1 埋め込みウィジェット

テナントのHTMLに1行追加するだけで設置できます。

```html
<script src="https://cdn.example.com/widget.js" data-tenant="xxx"></script>
```

### 6.2 UI仕様

| 項目 | 仕様 |
|---|---|
| 初期表示 | 画面右下に固定ボタン（チャットアイコン） |
| 展開方法 | ボタンクリックでチャット画面を表示（モーダル or 別画面遷移） |
| 将来対応 | フローティング形式（画面上をドラッグで移動できる）を将来的に検討 |
| レスポンシブ | PC: 380×600px、SP: フルスクリーン |
| テーマ | テナント別カラー・ロゴをdata属性または設定ファイルから適用 |
| ストリーミング | SSEでトークンを逐次表示（タイピングアニメーション） |
| オフライン対応 | 接続エラー時にフォールバックメッセージを表示 |

### 6.3 メッセージUI

- テキストメッセージ：Markdownレンダリング対応（箇条書き・太字・リンク）
- 入力中インジケーター：LLM処理中はドットアニメーション表示
- スクロール制御：新メッセージで自動スクロール、ユーザーが上スクロール中は無効

> **設計方針**　ウィジェット内に電話・LINE・フォームへのリンクボタンは実装しません。

## 7. APIサーバ仕様

### 7.1 エンドポイント一覧

| メソッド | パス | 概要 | 認証 |
|---|---|---|---|
| POST | /v1/chat/message | メッセージ送信・LLM応答（SSEストリーム） | APIキー |
| POST | /v1/chat/session | 新規セッション開始（会話履歴をメモリで管理） | APIキー |
| GET | /v1/widget/config | ウィジェット設定取得（テーマ・テナント設定） | APIキー |

### 7.2 チャットAPIリクエスト/レスポンス

#### リクエスト（POST /v1/chat/message）

```json
{
  "session_id": "string | null",
  "message": "string",
  "metadata": {
    "page_url": "string"
  }
}
```

#### SSEレスポンスイベント

| イベントtype | payload例 | 説明 |
|---|---|---|
| chunk | `{content: 'テキスト'}` | テキストチャンク（逐次配信） |
| done | `{session_id: '...'}` | 応答完了シグナル |
| error | `{code: '...', message: '...'}` | エラー通知 |

### 7.3 セッション管理

DBを持たないため、会話履歴はサーバのメモリ上で管理します。

- セッションIDをキーにしたオンメモリのDict構造
- 有効期限：最終アクセスから30分でセッション破棄
- サーバ再起動時に会話履歴はリセット（許容する）
- 将来的に永続化が必要になった場合はRedisを追加

### 7.4 認証・セキュリティ

| 対象 | 方式 | 詳細 |
|---|---|---|
| ウィジェット↔API | APIキー（X-API-Keyヘッダ）+ Origin検証 | テナントごとに発行 |
| CORS | テナント登録ドメインのみ許可 | ワイルドカード不可 |
| 通信 | 全通信TLS 1.3必須 | Let's Encrypt or 自社証明書 |
| レートリミット | APIサーバ側でシンプルなIP/APIキー制限 | Redisなしで実装可能な方式 |
