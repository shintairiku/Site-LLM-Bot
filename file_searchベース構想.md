# file_searchベース構想

## 背景

- 現在は OpenAI Responses API の `web_search` を主体に利用する
- `allowed_domains` によるドメイン制限は必要
- ただし `web_search` の `filters` はモデル互換性に依存する
- 将来的には、外部検索依存を減らし、より強い根拠保証を行いたい

## 現在方針

- 当面は `web_search` を主系として運用する
- `filters.allowed_domains` に対応するモデルを使う
- 候補は `gpt-5.4-mini`、必要に応じて `gpt-5.5`

## 将来方針

- 根拠データを自社管理のナレッジに寄せる
- OpenAI Responses API の `file_search` を主系に移行する
- 自社サイト由来のテキストのみを vector store に投入し、その corpus のみを検索対象にする

## `file_search` 化の狙い

- 回答根拠を自社サイト由来データに限定しやすい
- `web_search` より外部情報混入リスクを下げられる
- 検索対象を安定化できる
- クロール済みコンテンツを使うため、外部サイトの瞬断や検索結果変動の影響を受けにくい

## 想定アーキテクチャ

1. 自社サイトを定期クロールする
2. ページ本文を整形し、不要要素を除去する
3. 文書を OpenAI Files / Vector Store に登録する
4. テナントごとに vector store を分ける
5. `/api/chat` から `file_search` を使って回答生成する
6. 必要なら `web_search` は補助系としてのみ使う

## 必要な実装要素

- クロール対象URL一覧の管理
- HTML から本文・見出し・URL・更新日時を抽出する前処理
- 文書の差分更新
- Vector Store へのアップロード処理
- テナントと vector store ID の対応管理
- `file_search` を使う Responses API 呼び出し実装
- 回答に使われた検索結果の監査方法

## データ設計メモ

- 1ページごとに元URLを保持する
- タイトル、見出し、本文、更新日時などを metadata として持たせる
- テナントIDを metadata に含める
- 公開停止ページを削除または無効化できるようにする

## 検討事項

- クロール頻度をどうするか
- robots.txt や認証付きページをどう扱うか
- 画像主体ページや PDF をどう取り込むか
- 同一内容の重複チャンクをどう抑制するか
- 回答で citation 表示をどう見せるか
- `web_search` を完全停止するか、フェイルオーバーとして残すか

## 移行ステップ案

1. まずは `web_search` で運用継続
2. 別系統でクロール・整形・投入バッチを作る
3. 小規模テナントで `file_search` を試験導入する
4. 回答品質と根拠の安定性を比較する
5. 問題なければ `file_search` を主系へ昇格する

## 判断メモ

- 「対象サイトの情報だけを根拠にしたい」という要件に対しては、長期的には `file_search` の方が設計的に自然
- 一方で初期構築コストは `web_search` より高い
- そのため、短期は `web_search`、中長期は `file_search` 主体が現実的
