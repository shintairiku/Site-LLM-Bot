# AIチャットボット v2.0 コードレビュー調査結果

調査日: 2026-05-12

最終更新: 2026-05-13

## 調査目的

`AI_Chatbot_仕様書_v2.0.md` を前提に、現在のリポジトリ構成と実装状況を確認し、販売先へ以下を配布する想定でコードレビューを行った。

- スクリプトタグ
- ウィジェット JavaScript
- ウィジェット CSS

## 仕様書から読み取ったプロジェクト概要

本プロジェクトは、住宅・リフォーム業向けホームページに埋め込む AI チャットボットである。DB や事前インデックスを持たず、リクエスト時にテナント HP 限定の Web Search を行い、その結果を根拠として LLM が回答するシンプルな構成を目指している。

主な要件は以下。

- テナント HTML に 1 行の script タグで設置する
- 右下固定ボタンからチャット画面を開く
- PC は 380 x 600px 程度、SP はフルスクリーン表示
- LLM 応答は SSE ストリーミングで逐次表示する
- API は `/v1/chat/message`、`/v1/chat/session`、`/v1/widget/config` を提供する
- API は API キーまたは公開トークン相当の認証と Origin 検証を行う
- CORS はテナント登録ドメインのみ許可し、ワイルドカードは禁止する
- 会話履歴は初期段階ではオンメモリ管理、TTL は 30 分
- 検索対象はテナントごとの `allowed_domains` に限定する

## 重大度別レビュー

### High: 認証・Origin 検証が未実装

仕様では、ウィジェットと API の間で API キーまたは公開トークン相当の認証と Origin 検証が必要である。しかし現在は `tenant_id` をリクエストボディで渡すだけで `/api/chat` を呼び出せる。

さらに `CORSMiddleware` が `allow_origins=["*"]` になっており、任意 Origin から呼び出し可能である。

影響:

- 第三者が API を直接呼び出せる
- LLM API コストが不正利用される
- テナントなりすましが可能になる
- 仕様の「ワイルドカード不可」に反する

該当箇所:

- `site_llm_bot/api/app.py`
- `config/tenants.json`

推奨対応:

- `TenantConfig` に `allowed_origins`、`allowed_origin_patterns`、`public_token`、`status` を追加する
- `X-Tenant-Id` と `X-Widget-Token` を検証する依存関数を追加する
- `Origin` がテナント設定に含まれる場合のみ許可する
- 本番環境では `null` Origin を拒否する
- CORS のワイルドカード許可を廃止する

対応状況:

- `fix/auth-origin-guard` で対応済み
- CORS はテナント設定の `allowed_origins` / `allowed_origin_patterns` から構築する形へ変更
- 実リクエストでは `X-Tenant-Id`、`X-Widget-Token`、`Origin` を検証
- Vercel dev / Cloud Run dev の動作確認で不足していた Origin を追加
- デモページが古い本番 Cloud Run の JS を固定参照していたため、`/static/mock-widget.js` 参照へ修正

### High: 仕様上の `/v1` API が未実装

仕様書では以下の API が定義されている。

- `POST /v1/chat/message`
- `POST /v1/chat/session`
- `GET /v1/widget/config`

現状は `/api/chat` のみで、レスポンスも JSON 一括返却である。

影響:

- 販売用ウィジェットと API 契約が固まっていない
- セッション開始、設定取得、メッセージ送信の責務が分離されていない
- 本番用 API に移行する際にウィジェット側の破壊的変更が必要になる

推奨対応:

- `/api/chat` はデモ互換として残し、本番用に `/v1/*` を追加する
- `/v1/widget/config` で表示名、テーマカラー、挨拶、候補質問を返す
- `/v1/chat/session` でセッションを明示発行する
- `/v1/chat/message` にチャット送信処理を移す

### High: SSE ストリーミング未実装

仕様では LLM 応答を SSE で逐次返却し、ウィジェット側でタイピング表示する想定である。現状は FastAPI 側が通常 JSON を返し、ウィジェット側も `fetch()` で完了応答を待つ。

影響:

- 初回表示までの体感速度が悪くなる
- 仕様の SSE レスポンスイベント `chunk` / `done` / `error` に対応できない
- LLM 応答中の UX が弱い

推奨対応:

- FastAPI 側に `StreamingResponse` または SSE 用レスポンスを追加する
- OpenAI Responses API の streaming に対応したハンドラを追加する
- ウィジェット側で `ReadableStream` または `EventSource` 相当の処理を実装する
- イベント形式を `chunk`、`done`、`error` に統一する

### High: 配布用ウィジェットとして未整理

現在の配布対象は `mock-widget.js` / `mock-widget.css` という名前であり、Cloud Run の URL がハードコードされている。候補質問、挨拶、テナント名も JS 内に初期値として入っている。

販売先へ配る前提では、静的アセットは共通化し、テナント差分は script タグの `data-*` と API 設定取得で吸収する必要がある。

影響:

- テナント追加時に JS を編集する運用になりやすい
- ステージング / 本番 URL の切り替えが不安定になる
- 販売先へ「mock」という名前のファイルを配布することになる

推奨対応:

- `widget.js`、`widget.css` へ名称を整理する
- script タグは `data-tenant-id` と `data-public-token` のみに近づける
- `data-api-base` は開発・検証用途に限定する
- 表示名、挨拶、候補質問、テーマカラーは `/v1/widget/config` から取得する
- CDN / 静的配信向けにキャッシュ方針とバージョンパスを決める

想定 script タグ:

```html
<script
  src="https://widget.example.com/widget.v1.js"
  data-tenant-id="tenant-abc"
  data-public-token="public_xxxxx"
  defer
></script>
```

### Medium: `allowed_origins` が設定ファイルにあるが使われていない

`config/tenants.json` には `allowed_origins` と `allowed_origin_patterns` が定義されている。しかし `TenantConfig` には該当フィールドがなく、読み込み・検証に使われていない。

影響:

- 設定ファイルと実装の意味がズレている
- ドキュメント上は Origin 制御しているように見えるが実際には無効
- レビューや運用時に誤解を招く

推奨対応:

- `TenantConfig` に Origin 関連フィールドを追加する
- Origin 検証ロジックをテストする
- 現状と矛盾するテストを修正する

対応状況:

- `fix/auth-origin-guard` で対応済み
- `TenantConfig` が `allowed_origins`、`allowed_origin_patterns` を読み込むように変更
- 未登録 Origin を拒否し、登録 Origin を許可するテストを追加

### Medium: セッションがテナントに紐づいていない

現在のセッションストアは `session_id` をキーに履歴を保持している。テナント ID や Origin と紐づいていないため、同一 `session_id` を別テナント指定で送ると履歴混在の余地がある。

影響:

- テナント間で会話履歴が混ざる可能性がある
- 認証追加後にセッション整合性の検証が必要になる

推奨対応:

- `ChatSession` に `tenant_id`、`origin`、`page_url` を持たせる
- `get_or_create` 時にテナント不一致を拒否する
- `/v1/chat/session` でセッションを作成し、以降は同じテナントでのみ使えるようにする

### Medium: レートリミットが未実装

仕様では Redis なしでも実装可能な IP / API キー単位のレートリミットが求められている。現状は無制限である。

影響:

- LLM API コストが急増する
- 悪意ある連投や誤実装に弱い
- 同時接続 30 セッション / テナントの運用目標を守りにくい

推奨対応:

- 初期実装としてプロセス内メモリで `tenant_id + ip` 単位の制限を入れる
- `/v1/chat/message` は 1 分あたり 20 回程度から開始する
- 将来 Redis へ差し替えやすい interface を用意する

### Low: テストが販売仕様ではなく現実装を固定している

現在のテストには、CORS が任意 Origin を許可することを期待するケースがある。これは販売前提の仕様と逆である。

影響:

- 仕様準拠へ修正したときにテストが失敗する
- セキュリティ上の問題をテストが肯定してしまう

推奨対応:

- 未登録 Origin を拒否するテストへ変更する
- `X-Tenant-Id`、`X-Widget-Token`、`Origin` の組み合わせ検証を追加する
- `/v1/widget/config`、`/v1/chat/session`、SSE イベント形式のテストを追加する

## 配布物観点の評価

### スクリプトタグ

現状はデモページ内で以下に近い埋め込みになっている。

```html
<script
  id="site-llm-bot-widget"
  src="/static/mock-widget.js"
  data-tenant-id="sample-shintairiku"
  data-tenant-name="新大陸"
  data-public-token="public_sample_shintairiku"
  data-color="#155e75"
  data-api-base="__WIDGET_API_BASE__"
></script>
```

販売用としては、`data-tenant-name` や `data-color` を顧客サイトに持たせず、サーバー側設定から取得する方がよい。顧客に渡す script タグは、テナント ID と公開トークン程度に抑える。

### ウィジェット JavaScript

良い点:

- 自己実行関数でグローバル汚染を抑えている
- CSS を自動読み込みしており、script タグだけでも動く
- DOM 追加時に外部入力を text node 化しており、基本的な XSS 配慮がある
- gzip サイズは小さい

改善点:

- `mock-widget.js` から正式名称へ変える
- API URL のハードコードをなくす
- `/api/chat` ではなく `/v1/*` を呼ぶ
- 設定取得 API を使う
- SSE ストリーミングを処理する
- API 失敗時にモック応答を返す仕様は本番では無効化する

### ウィジェット CSS

良い点:

- UI は軽量で、右下固定ボタンとパネル表示に対応している
- SP ではフルスクリーンになる
- CSS サイズは小さい

改善点:

- `mock-chatbot-*` という class prefix を正式 prefix に変える
- テナントサイトの CSS と衝突しにくい命名にする
- Shadow DOM 採用も検討する
- テーマ変数は `/v1/widget/config` の結果で反映する

## 結論

現在の実装は、仕様書 v2.0 の MVP 方針に沿ったプロトタイプとしては成立している。特に、Web Search の `allowed_domains` をテナントごとに切り替える導線、オンメモリセッション、軽量ウィジェットは実装済みである。

一方で、販売先へ「スクリプトタグ」「ウィジェット JavaScript」「ウィジェット CSS」を配布する前提では、SSE、`/v1` API、設定取得 API が未整備である。認証・Origin 制御は `fix/auth-origin-guard` で対応済みのため、次の優先事項は本番用 API 契約の確定である。
