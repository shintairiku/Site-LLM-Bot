# AIチャットボット サーバー設計

## 1. 前提

- 最終的に販売するのは埋め込み用ウィジェットスクリプト
- バックエンドは自社サーバーで集約運用する
- テナントごとに埋め込み先ドメイン、テーマ、システムプロンプト、検索対象ドメインを分離する
- この段階では OpenAI API の実疎通確認は行わない

## 2. 設計方針

- ウィジェット配布サーバーと API サーバーを論理的に分離する
- テナント識別は `tenant_id` を中心に行い、機密情報は埋め込みHTMLへ直接置かない
- ウィジェットから API へは `tenant_id + 署名付き公開トークン + Origin 検証` で接続する
- 会話履歴は初期段階では API サーバーのメモリ管理でよいが、将来 Redis へ差し替えやすい構造にする
- OpenAI / Claude などの LLM 呼び出しはアプリ層の抽象化に閉じ込め、ウィジェットからは隠蔽する

## 3. 推奨構成

```text
[ユーザーのブラウザ]
  └─ <script src="https://widget.example.com/widget.js" ...>
       ↓
[Widget CDN / Static Server]
  └─ widget.js, widget.css を配布
       ↓ HTTPS
[API Gateway / Nginx]
  ├─ TLS終端
  ├─ Origin / Rate Limit / Access Log
  └─ /v1/* を FastAPI へ中継
       ↓
[FastAPI App]
  ├─ テナント設定取得
  ├─ セッション管理
  ├─ 入力検証
  ├─ LLM呼び出し
  └─ SSE配信
       ↓
[Session Store]
  ├─ 初期: メモリ
  └─ 将来: Redis
       ↓
[LLM Provider]
  └─ OpenAI / Anthropic / xAI
```

## 4. 責務分離

### 4.1 Widget 配布サーバー

- `widget.js`
- `widget.css`
- 必要なら画像やフォント

責務:
- テナントサイトへ配る静的資産の高速配信
- キャッシュ制御
- バージョン配信

非責務:
- 会話処理
- APIキー保持
- テナント秘密情報の配布

### 4.2 API サーバー

責務:
- テナント認証
- ウィジェット設定返却
- セッション発行
- チャット受付
- LLM 呼び出し
- SSE 応答
- アクセス制御とログ出力

### 4.3 管理データ層

初期段階はファイルまたは環境変数で開始してよいですが、販売前提なら最終的には最低限のテナント管理テーブルが必要です。

必要データ:
- `tenant_id`
- `tenant_name`
- `status`
- `allowed_origins`
- `allowed_domains`
- `widget_theme`
- `public_token`
- `server_side_secret`
- `model_provider`
- `model_name`
- `system_prompt`

## 5. 埋め込み方式

販売形態を踏まえると、最終的な埋め込み形式は以下を推奨します。

```html
<script
  src="https://widget.example.com/widget.v1.js"
  data-tenant-id="tenant-abc"
  data-public-token="public_xxxxx"
  defer
></script>
```

理由:
- テナントごとの秘密鍵を HTML に置かずに済む
- サーバー側で `tenant_id` と `public_token` の組み合わせを検証できる
- CDN 上のスクリプトを共通化できる

## 6. 認証・認可設計

### 6.1 現段階で避けるべきこと

- OpenAI API Key をブラウザへ渡す
- テナント秘密鍵を `data-*` に直接埋め込む
- CORS を `*` にする

### 6.2 推奨方式

ウィジェット -> API:
- `X-Tenant-Id`
- `X-Widget-Token`
- `Origin`

サーバー側検証:
1. `tenant_id` が有効か
2. `widget_token` がそのテナントに対応するか
3. `Origin` が登録ドメイン内か
4. 停止済みテナントではないか

補足:
- `widget_token` は公開前提の識別子でよく、単独では強い秘密にしない
- 強い秘密はサーバー側 `server_side_secret` として保持し、署名検証や内部API連携に使う

## 7. API 設計

販売前提なら、フロント確認用の `/api/chat` ではなく、以下の本番系パスに寄せるべきです。

### 7.1 `GET /v1/widget/config`

用途:
- テーマ
- 表示名
- 初期挨拶
- 候補質問

入力:
- `X-Tenant-Id`
- `X-Widget-Token`

### 7.2 `POST /v1/chat/session`

用途:
- セッション発行
- 初回検証

返却:
- `session_id`
- `expires_in`

### 7.3 `POST /v1/chat/message`

用途:
- ユーザー発話送信
- SSE ストリーミング返却

入力:
- `session_id`
- `message`
- `metadata.page_url`

返却:
- `chunk`
- `done`
- `error`

## 8. セッション設計

初期:
- FastAPI プロセス内メモリ
- TTL 30分
- 直近数件のみ保持

販売後の推奨:
- Redis へ移行

理由:
- 複数 API インスタンスへ水平分割しやすい
- Nginx 配下で複数ワーカー運用しやすい
- サーバー再起動で会話が消える問題を緩和できる

セッションに持つもの:
- `session_id`
- `tenant_id`
- `messages`
- `created_at`
- `updated_at`
- `page_url`

## 9. CORS / Origin 設計

販売用では以下を必須にします。

- CORS はテナント登録済み Origin のみ許可
- `null` Origin は開発時専用
- 本番では `https://` のみ許可
- `Origin` ヘッダが欠けるケースはブラウザ以外として別扱いにする

開発環境:
- `http://127.0.0.1:*`
- `http://localhost:*`

本番環境:
- テナント管理データに登録した URL のみ

## 10. レートリミット

初期実装:
- `tenant_id + IP` 単位で分単位制限
- `session_id` 単位でも短時間連投を制限

例:
- `/v1/chat/message`: 1分あたり 20 回
- `/v1/chat/session`: 1分あたり 10 回

販売前提では、少なくとも以下をログに残します。
- tenant_id
- origin
- ip
- path
- status_code
- latency_ms

## 11. ログと監視

最低限必要:
- リクエストログ
- エラーログ
- LLM 呼び出し時間
- Provider 別失敗率

将来追加:
- テナント別利用量
- 日次会話数
- 平均応答時間
- トークン消費量

## 12. デプロイ構成

### 初期MVP

- `nginx`
- `fastapi`

### 販売前に追加推奨

- `redis`
- `postgresql` または軽量な設定DB
- `worker`（非同期処理が増えた場合）

## 13. 実装上の指針

### 今のコードから見た変更方針

1. `sample_page` 用の暫定 `/api/chat` は残してよいが、本番用は `/v1/...` へ寄せる
2. `data-api-base` だけでなく `data-tenant-id` と `data-public-token` を受ける構造へ変える
3. セッション管理は `session_store` 抽象を維持し、Redis 実装を差し替え可能にする
4. `allowed_origins` は固定 env ではなくテナント設定から引く
5. ウィジェット設定取得APIを用意し、色・文言を HTML ハードコードから外す

## 14. 現時点の結論

販売用ウィジェットを前提にすると、サーバーは以下の3層で設計するのが妥当です。

- 配布層: CDN または静的サーバーで `widget.js` を配布
- API層: 自社 FastAPI サーバーで認証、会話、LLM 呼び出しを担当
- 状態管理層: 初期はメモリ、販売前に Redis へ移行

また、本番前提では「ウィジェットは公開情報だけを持つ」「秘密は必ず自社サーバーに置く」という原則を崩さないことが重要です。
