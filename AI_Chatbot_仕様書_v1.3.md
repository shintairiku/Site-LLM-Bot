## 1. システム概要

### 1.1 プロダクトビジョン

本システムは、住宅・リフォーム業の顧客企業（工務店・リフォーム店等）向けに提供するSaaS型AIチャットボットプラットフォームです。

#### チャットボットの位置づけ

チャットボットは顧客企業のホームページに設置される集客導線（CTA）のひとつです。既存のLINEボタン・電話ボタンと並んで新たな導線として追加されます。チャットボット内でLINEや電話へ誘導するボタンを生成する機能は持ちません。

| 導線種別 | 役割 | 本システムの対象 |
|---|---|---|
| 電話ボタン | 即時の問い合わせ・商談 | 対象外（既存UI） |
| LINEボタン | 継続的なコミュニケーション | 対象外（既存UI） |
| AIチャットボット | 24時間の疑問解消・ヒアリング支援 | ★ 本システム |

> **設計方針**　チャットボットの役割はユーザーの疑問を解消し、信頼を醸成することです。問い合わせへの誘導は既存導線に委ねます。

### 1.2 システム構成レイヤー

- フロントエンド層：埋め込みウィジェット（iframe / Web Component / スクリプトタグ）
- APIゲートウェイ層：テナント認証・レート制限・ルーティング
- AIエンジン層：LangGraph自律エージェント + ハイブリッドRAG + Web Search Tool
- データ層：ベクトルDB（Qdrant）・RDB（PostgreSQL）・セッションキャッシュ（Redis）
- オブザーバビリティ層：Langfuse（トレーシング）+ RAGAS（評価）

### 1.3 主要機能サマリ

| 機能カテゴリ | 概要 | 優先度 |
|---|---|---|
| チャットUI | 埋め込みウィジェット・独自UI・iframeサポート | 最優先 |
| 自律AIエージェント | LangGraph + LLMによるツール自律選択・応答生成 | 最優先 |
| ハイブリッドRAG | BM25 + HNSW検索 + Re-rankerの三段構え | 最優先 |
| Web Search Tool | 検索ドメイン指定付きWeb検索（RAGの補完） | 最優先 |
| マルチテナント | テナント分離（設計オプション選択式、後述） | 最優先 |
| オブザーバビリティ | Langfuseによるトレーシング・プロンプト管理 | 高 |
| 管理ダッシュボード | 会話ログ閲覧・プロンプト編集・ウィジェット設定 | 高 |
| RAG評価 | RAGASによる検索・生成品質の定量評価 | 中（Phase 4） |

## 2. アーキテクチャ設計

### 2.1 技術スタック

> **方針**　バックエンドはPython / Node.js（TypeScript）どちらでも対応可能な設計とします。以下はPython（FastAPI）採用時の推奨構成です。

| レイヤー | 技術選定 | 備考 |
|---|---|---|
| フロントエンド | TypeScript + Preact / Vanilla JS | 軽量ウィジェット ≤50KB（gzip） |
| バックエンドAPI | Python 3.12 + FastAPI | 非同期・SSEストリーミング対応 |
| AIオーケストレーション | LangGraph（自律エージェント） | ToolNode + ReActループ |
| LLMプロバイダ | Anthropic Claude（主）/ OpenAI（副） | 抽象レイヤーで切替可能 |

| ハイブリッド統合 | RRF（Reciprocal Rank Fusion） | Qdrant内蔵。BM25とベクトルスコアを統合 |
| Re-ranker | CrossEncoder（BGE-reranker-v2） | ハイブリッド検索後の精密再順位付け |
| RDB | PostgreSQL 16 | テナント・会話・設定管理 |
| キャッシュ/セッション | Redis 7 | セッション・レートリミット・検索キャッシュ |
| クロール/インデックス | Crawl4AI + カスタムパイプライン | 定期実行・差分更新 |
| オブザーバビリティ | Langfuse（セルフホスト） | トレーシング・プロンプト管理・評価ログ |
| RAG評価 | RAGAS + LangSmith | 検索・生成品質の定量評価（Phase 4） |
| インフラ | Docker Compose（初期）→ K8s移行可 | 自社サーバ前提 |

### 2.2 マルチテナント設計オプション

テナント管理方式は現在検討中のため、以下2案を比較提示します。実装着手前に確定が必要です。

| 比較項目 | Option A：論理分離（tenant_id） | Option B：サブドメイン分離 |
|---|---|---|
| 概要 | 単一DBでtenant_idで論理分離 | テナントごとにサブドメイン+スキーマ/DB分離 |
| 初期開発コスト | 低い | 高い |
| テナント追加 | DB操作のみ・即時 | DNS/証明書設定が必要 |
| データ分離強度 | RLS + アプリ層制御 | DB/スキーマ単位で強固 |
| スケーラビリティ | テナント増加で単一DBが肥大化 | テナント単位でスケール可能 |
| 推奨フェーズ | MVP〜テナント50社規模まで | 本格展開後・エンタープライズ向け |

> **推奨**　MVP段階ではOption A（論理分離）で開発着手し、テナント数・要件に応じてOption Bへ移行するハイブリッド戦略を推奨します。

### 2.3 ドメインプロバイダ構成

- ウィジェット埋め込みスクリプトにはテナント別APIキーを含める
- APIゲートウェイはX-API-KeyヘッダとOriginの組み合わせでテナントを特定
- Option B採用時：`chat.koumuten-abc.co.jp` のようなサブドメインでテナントを識別

## 3. AIエンジン仕様（LangGraph 自律エージェント）

### 3.1 設計方針

LangGraphのReActループをベースに、LLMが状況に応じてツール（ハイブリッドRAG検索・Web検索）を自律的に選択・実行し、ユーザーの疑問に対して正確な回答を生成します。エージェントの最終出力は常にテキスト回答のみです。

### 3.2 実行時ワークフロー

| ステップ | 処理 | 担当 |
|---|---|---|
| ① ユーザー質問受信 | チャットUIからメッセージを受け取りセッションに追加 | APIサーバ |
| ② エージェント起動 | LangGraphがReActループを開始し、使用ツールを自律判断 | LangGraph |
| ③ BM25検索（キーワード） | 固有名詞・地名・工法名などの完全一致クエリに対応 | rag_searchツール |
| ④ HNSW検索（意味検索） | クエリの意図・同義語を捉えた近傍探索。上位20件取得 | rag_searchツール |
| ⑤ RRFで統合 | BM25とHNSWの結果をReciprocal Rank Fusionで一本化 | rag_searchツール |
| ⑥ Re-rank（精密絞り込み） | CrossEncoderで統合結果を精密スコアリング。上位5件に絞る | rag_searchツール |
| ⑦ Web検索（必要な場合） | RAGで十分な情報が得られない場合、許可ドメイン限定でWeb検索を実行 | web_searchツール |
| ⑧ 回答生成 | 絞り込んだコンテキストを元にLLMがテキスト回答を生成・ストリーミング返却 | LLM（Claude） |
| ⑨ トレース記録 | 全ノードの入出力・レイテンシをLangfuseへ非同期送信 | Langfuse SDK |
| ⑩ ログ保存 | 会話メッセージをPostgreSQLへ非同期で永続化 | loggerノード |

> **重要**　エージェントの最終出力はテキスト回答のみです。CTAボタン・LINE誘導・電話促進などのUI要素は生成しません。

### 3.3 エージェントグラフ構成

#### ノード定義

| ノード名 | 種別 | 処理内容 |
|---|---|---|
| agent | LLMノード | ユーザー入力を受け取りツール選択・応答生成を行うReActコアノード |
| rag_search | Toolノード | BM25 + HNSW検索 → RRF統合 → Re-rankerの三段構えでコンテキスト取得 |
| web_search | Toolノード | テナント許可ドメイン限定のWeb Search Tool実行 |
| memory_read | Toolノード | セッション内の会話履歴・収集済み情報の参照 |
| logger | Sinkノード | 会話ログをPostgreSQLへ永続化（非同期） |

#### エッジ（遷移）設計

- `agent → ToolNode群`：LLMのtool_callsに基づいて動的にルーティング
- `ToolNode → agent`：ツール結果をLLMに返してReActループを継続
- `agent → END`：tool_callsが空（最終テキスト回答生成完了）で終了
- 最大ループ回数：10回（無限ループ防止。超過時はエラーメッセージを返す）

### 3.4 ツール定義仕様

| ツール名 | 入力スキーマ | 出力 |
|---|---|---|
| rag_search | `query: str, top_k: int = 5` | `List[{content, url, score, retrieval_method}]` |
| web_search | `query: str, domains: List[str]` | `List[{title, url, snippet}]` |
| get_session_context | `session_id: str` | `Dict`（会話履歴・収集済み情報） |

### 3.5 システムプロンプト設計

| プロンプト区分 | 管理者 | 主な内容 |
|---|---|---|
| グローバル基盤 | プラットフォーム管理者 | 安全性ガイドライン・ツール使用判断基準・応答フォーマット・ハルシネーション防止・回答はテキストのみとする指示 |
| 業種共通 | プラットフォーム管理者 | 住宅業界の専門用語対応・リフォーム/新築の典型質問パターン・丁寧語設定 |
| テナント固有 | テナント管理者（ダッシュボードから編集） | 会社名・対応エリア・強み・NGワード・トーン・Web検索対象ドメイン |

## 4. RAGパイプライン（ハイブリッド検索）

### 4.1 ハイブリッド検索の設計根拠

純粋なベクトル検索（HNSW）は意味的な近傍探索に優れますが、固有名詞・地名・工法名・商品コードなどの完全一致クエリで精度が落ちます。BM25はその逆で、同義語や言い換えを見逃します。住宅・リフォーム業では工務店名・地域名・工法名を含む質問が多いため、両者を組み合わせるハイブリッド検索が本ユースケースに適しています。

| 手法 | 得意なクエリ例 | 苦手なクエリ例 |
|---|---|---|
| BM25（キーワード） | 「〇〇工務店の施工事例」「羽島市 リフォーム」 | 「家が古くなってきた」「暖かい家にしたい」 |
| HNSW（ベクトル） | 「断熱性能を上げたい」「快適な住環境」 | 「UA値0.46」「ZEH基準」（専門用語の完全一致） |
| ハイブリッド（両方） | 両方のクエリパターンをカバー | どちらかが補完するため失敗しにくい |

### 4.2 インデックス構築フロー（事前準備）

| ステップ | 処理 | 技術/パラメータ |
|---|---|---|
| 1. クロール | Crawl4AIでサイト全体をスクレイピング | robots.txt準拠・JS動的ページ対応 |
| 2. チャンキング | ページをセクション単位で分割 | 512トークン・オーバーラップ64 |
| 3. エンベディング | テキストをベクトル化（密ベクトル） | text-embedding-3-small / bge-m3（ローカル） |
| 4. BM25インデックス | テキストを疎ベクトル化してQdrantへ登録 | Qdrantのsparse vectorsを使用 |
| 5. HNSWインデックス | 密ベクトルをQdrantへアップサート | m=16, ef_construction=200 |
| 6. メタデータ付与 | URL・タイトル・最終更新日をペイロードに保存 | テナント別フィルタリングに使用 |

> **実装**　QdrantはBM25（sparse vector）とHNSW（dense vector）を同一コレクションで管理できます。別途BM25ライブラリの追加は不要です。

### 4.3 検索時フロー（rag_searchツール内）

#### 三段構えの検索アーキテクチャ

| 段階 | 手法 | 役割 | 出力件数 |
|---|---|---|---|
| 第1段：並列検索 | BM25（キーワード）+ HNSW（ベクトル）を同時実行 | 広くキャンディデートを取得 | 各20件 |
| 第2段：RRF統合 | Reciprocal Rank Fusionで2つのランキングを一本化 | 補完的な結果を公平に統合 | 20件 |
| 第3段：Re-rank | CrossEncoder（BGE-reranker-v2）で精密スコアリング | 本当に関連する文書だけを残す | 上位5件 |

#### RRF（Reciprocal Rank Fusion）について

RRFはBM25とHNSWそれぞれのランキング順位をスコアに変換し、単純加算で統合するアルゴリズムです。スコアの正規化が不要で、実装がシンプルかつ精度が高い点から業界標準となっています。Qdrantに内蔵されているため、追加実装は不要です。

### 4.4 Web Search Tool（ドメイン制限付き）

Anthropic Tool UseのWeb Search Toolを使用し、テナントが許可したドメインのみを検索対象とします。RAGで情報が不足している場合のフォールバックとして機能します。

- `tenant_domains`テーブルにテナントごとのホワイトリストを保存
- エージェントはweb_searchツール呼び出し時に`allowed_domains`を自動付与
- 検索結果はRedisでキャッシュ（TTL 1時間）してAPIコストを削減

> **参考**　ドメイン制限実装は添付gitリポジトリのWeb Search Tool実装パターンに準拠します。

## 5. フロントエンド仕様

### 5.1 埋め込みウィジェット

#### 配布形式（3方式をすべてサポート）

```html
<!-- スクリプトタグ形式（推奨） -->
<script src="https://cdn.example.com/widget.js" data-tenant="xxx"></script>

<!-- iframe形式 -->
<iframe src="https://chat.example.com/widget?tenant=xxx"></iframe>

<!-- Web Component形式 -->
<ai-chatbot tenant="xxx"></ai-chatbot>
```

| 項目 | 仕様 |
|---|---|
| バンドルサイズ | 初期ロード ≤50KB（gzip）、AIエンジン部分は遅延ロード |
| 表示形式 | 右下固定ボタン → クリックで展開（モーダル / サイドパネル） |
| レスポンシブ | PC: 380×600px、SP: フルスクリーン |
| テーマ | テナント別カラー・ロゴ・フォントをAPIから取得して動的適用 |
| ストリーミング | SSEでトークンを逐次表示（タイピングアニメーション） |
| オフライン対応 | 接続エラー時にフォールバックメッセージを表示 |

### 5.2 メッセージUI仕様

- テキストメッセージ：Markdownレンダリング対応（見出し・箇条書き・太字・リンク）
- クイックリプライ：エージェントが生成した選択肢ボタン（最大5個）
- カード形式：施工事例・プランを画像+タイトル+説明でカード表示（RAG結果活用）
- 入力中インジケーター：エージェント処理中はドットアニメーション表示
- スクロール制御：新メッセージで自動スクロール、ユーザーが上スクロール中は無効

> **設計方針**　ウィジェット内に電話・LINE・フォームへのリンクボタンは実装しません。テキスト回答の品質がチャットボットの唯一の価値提供手段です。

## 6. データベース設計

### 6.1 主要テーブル一覧

| テーブル名 | 概要 | 主要カラム |
|---|---|---|
| tenants | テナントマスタ | id, name, domain, api_key, system_prompt, config_json, created_at |
| tenant_domains | Web検索許可ドメイン | id, tenant_id, domain, is_search_allowed |
| conversations | 会話セッション | id, tenant_id, session_id, page_url, started_at, ended_at |
| messages | メッセージログ | id, conversation_id, role, content, tool_calls_json, retrieval_method, created_at |
| documents | RAGインデックス元文書 | id, tenant_id, url, title, content, embedding_id, bm25_indexed, indexed_at |
| users | 管理ダッシュボードユーザー | id, tenant_id, email, role, password_hash, last_login_at |

> **変更点**　messagesテーブルに`retrieval_method`カラムを追加（bm25 / hnsw / hybrid / web）。どの検索手法が使われたかをトレース可能にします。

### 6.2 Row Level Security（RLS）

PostgreSQL RLSを全テーブルに適用し、アプリバグによるテナント間データ漏洩を防止します。

- アプリ接続ロール：`app_user`（一般）・`admin_user`（管理）の2種を分離
- ポリシー：`app_user`は自テナントの行のみSELECT/INSERT/UPDATE/DELETE可能
- `tenant_id`はJWTクレームから取得し、`SET LOCAL app.tenant_id = '...'` で接続に注入

## 8. オブザーバビリティ・評価

### 8.1 Langfuse（トレーシング・プロンプト管理）

LangfuseはLLMアプリケーション向けのOSSオブザーバビリティプラットフォームです。自社サーバへのセルフホストが可能なため、会話データを外部送信せずに詳細なトレーシングを実現できます。

| 機能 | 詳細 |
|---|---|
| トレーシング | LangGraphの全ノード（agent / rag_search / web_search等）の入出力・レイテンシを自動記録 |
| プロンプト管理 | システムプロンプトのバージョン管理・A/Bテスト・ロールバック |
| 評価ログ | RAGASのスコアをLangfuseへ送信して一元管理 |
| デバッグ | RAG失敗時にどのチャンクが取得されたか・Re-rankerスコアを詳細追跡 |
| セルフホスト | Docker Composeで自社サーバに展開。会話データの外部送信なし |

> **採用理由**　LangSmithはLangGraphとのシームレスな統合が強みですが、クラウド送信が前提です。Langfuseはセルフホスト可能なため、自社サーバ運用という本件の前提と相性が良く採用します。

### 8.2 RAGAS（RAG評価フレームワーク）

RAGASはRAGパイプラインの品質を定量評価するOSSフレームワークです。グラウンドトゥルース（正解データ）なしで評価できる点が特徴で、Phase 4の品質向上フェーズから導入します。

| 評価指標 | 内容 | 対象 |
|---|---|---|
| Context Precision | 取得チャンクのうち本当に関連するものの割合 | 検索品質 |
| Context Recall | 正解に必要な情報がチャンクに含まれている割合 | 検索品質 |
| Faithfulness | 回答がチャンクの内容から逸脱していないか | 生成品質 |
| Answer Relevancy | 回答がユーザーの質問に対して適切か | 生成品質 |

#### 評価ワークフロー

- 定期バッチ：週次で直近の会話ログをサンプリングしRAGASで評価
- 結果をLangfuseへ送信して時系列で可視化
- Context Precisionの低下 → チャンキング・Re-rankerパラメータの調整
- Faithfulnessの低下 → システムプロンプトの改善またはRAGコンテキスト拡張
